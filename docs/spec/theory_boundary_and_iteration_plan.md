# Theory boundary and iteration plan

Updated on 2026-05-22 after the conservative Scientific Data package review and the decision to pause Zenodo upload until manuscript strength is improved.

## Current thesis boundary

The current evidence supports a bounded mechanism claim:

- Structured terminal, process, filesystem, and time receptor fields can be converted into bounded native-head command-selection decisions.
- NSI latent signals contribute under semantic-required command-selection conditions when compared with no-NSI and text-only controls.
- The strongest completed evidence is still controlled: sealed semantic-required transfer, multi-model and multi-seed local reproduction, and public repository trace pressure.

The current evidence does not support an unrestricted architecture claim:

- It does not prove production autonomy.
- It does not prove open-ended software repair.
- It does not prove unrestricted shell synthesis.
- It does not prove superiority over modern coding-agent loops on long-horizon repair tasks.
- It does not prove independent external reproduction.
- It does not prove epoch-making architecture status.

## Three-paper separation

### Paper A: data descriptor

Purpose: publish reusable benchmark traces, manifests, checksums, audits, and validation records.

Allowed claim:

- The archive is a reusable dataset for bounded repository command-selection research.

Not allowed:

- Any architecture proof or production-agent claim.

Current status:

- Locally viable after the rejection-risk fixes.
- Still blocked by DOI or reviewer URL, code release identifier, author metadata, and license review.

### Paper B: bounded mechanism paper

Purpose: argue that the native nervous-interface mechanism improves controlled command selection.

Allowed claim if evidence remains consistent:

- Bounded native-head Debug Cortex command selection with NSI latent contribution under preregistered non-sealed and sealed-final tests.

Required before submission:

- Clean artifact table from Phase2M through Phase2R.
- Explicit negative-evidence accounting for failed Phase2I, Phase2K sealed strict gate, and Phase2L sealed transfer.
- No retroactive use of sealed failures as design feedback.
- Comparison against prompt-only, ReAct, no-NSI, native-head-only/no-cache, and continuation-only.

Not allowed:

- Open-ended debugging, production autonomy, or unrestricted tool-use claims.

### Paper C: architecture/generalization paper

Purpose: test whether the architecture generalizes beyond bounded command selection.

Allowed claim only if future gates pass:

- Native nervous-interface mechanisms help open-ended repository repair under sandboxed, auditable, preregistered conditions.

Required before claim upgrade:

- Phase2S open-ended repair benchmark.
- Modern coding-agent loop baseline.
- Real patch generation, test execution, rollback, and stop-condition metrics.
- Repository-disjoint holdout.
- Independent or separately provisioned reproduction.
- Safety and destructive-action gates.

## Next iteration target

The next scientific question should not be "can the current system get a higher sealed-v3 score?" That question is already saturated and risks overfitting the story. The next question is:

Can the same bounded native nervous-interface mechanism improve sandboxed open-ended repair where the agent must inspect evidence, edit code, run tests, and stop safely?

This is the correct next target because it attacks the remaining unsupported claims directly:

- Dynamic real-repo debugging.
- Patch synthesis.
- Rollback and sandbox safety.
- Stop-condition discipline.
- Comparison to stronger agent baselines.

## Phase2S minimum viable benchmark

The first Phase2S version should stay small enough for the local GPU but strong enough to falsify the claim.

Required task families:

- Import or dependency mismatch.
- Failing unit assertion with localized source edit.
- Stale test snapshot requiring intentional snapshot update.
- Configuration or environment-marker mismatch.
- Multi-file relation requiring changed-file and traceback evidence.

Required graded factors:

- Candidate command count: 2, 3, and 4.
- Evidence density: low, medium, and high.
- Repair depth: one edit, two edits, and stale-state refresh.
- Failure observability: direct traceback, indirect changed-file relation, and ambiguous same-intent command.

Required claim-upgrade controls before any "strong architecture" language:

- The modern coding-agent loop baseline must be operationally fixed before the full gate: model/provider, tool budget, context policy, retry policy, edit permissions, stop rule, and cost or command-budget accounting.
- The statistical decision rule must be preregistered: bootstrap or Wilson intervals, task-family and difficulty stratification, no claim upgrade on ties, and at least `0.10` margins on both task success and patch correctness.
- The safety decision rule must be quantitative: zero unauthorized writes, zero allowlist/state hallucinations, zero low-level Qwen calls, rollback success `1.0`, false-completion rate `0.0`, and safety non-inferiority versus the best baseline.
- Epoch-making architecture status remains blocked until Phase2S, public holdout, multi-seed, multi-model, independent reproduction, and unanimous read-only reviewer simulation all pass. SubAgent agreement is necessary for internal review discipline but never sufficient without the measured gates.

Required controls:

- Full package.
- Native-head-only/no-cache.
- No-NSI latent.
- Continuation-only.
- Prompt-only.
- ReAct.
- Modern coding-agent-style loop.

Required metrics:

- Task success.
- Patch correctness.
- Test pass rate.
- Number of commands.
- Number of edits.
- Rollback success.
- Unauthorized write count.
- Low-level Qwen calls.
- Allowlist or state hallucination.
- Stop-condition correctness.

## Stop rules

- If the full package does not beat the best baseline by the preregistered margin, freeze the result as bounded-mechanism evidence only.
- If safety or rollback fails, stop architecture-claim escalation even if task success is high.
- If any task-specific command, path, or patch template is hardcoded, invalidate the benchmark evidence.
- If sealed-v3 information influences Phase2S design, invalidate the claim-bearing interpretation.

## Practical local execution strategy

Use local GPU constraints deliberately:

- Start with 16 to 32 tasks for infrastructure smoke.
- Move to 128 tasks for non-sealed pressure validation.
- Only then run 512 or larger full validation if smoke separates from baselines.
- Prefer short context and bounded candidate sets before increasing model size.
- Use command-sleep monitoring only when explicitly requested.
